## 二、已完成工作详情

### 2.1 打卡服务 — [services/checkinService.ts](backend/src/services/checkinService.ts)

**核心方法**：

| 方法 | 说明 |
|------|------|
| `doCheckin(userId, action)` | 执行打卡：一天一限 → 创建记录 → 计算连续天数 → 更新 User → 检查徽章 |
| `getCheckinHistory(userId, month?)` | 打卡历史（支持 YYYY-MM 月份筛选） |
| `getCheckinStreak(userId)` | 向前追溯连续天数（今天未打卡则从昨天算起） |
| `getBadges(userId)` | 已获得徽章列表 |
| `getCheckinStatus(userId)` | 打卡状态概览：todayCheckedIn / streak / totalCheckins / badges |
| `getCalendarData(userId, year, month)` | 日历月视图：`{ "2026-06-01": ["写代码", "做运动"], ... }` |
| `checkAndAwardBadges(userId)` | 🏅 徽章自动判定引擎（7种规则） |

**连续天数计算（`calcStreak`）**：

```
从今天开始向前追溯：
  Day 0 (今天):   有打卡 ✅ → streak = 1, 继续
  Day -1 (昨天):  有打卡 ✅ → streak = 2, 继续
  Day -2 (前天):  有打卡 ✅ → streak = 3, 继续
  Day -3:         无打卡 ❌ → 停止，返回 streak = 3
```

**幂等打卡设计**：
- 同一天第二次打卡 → 返回已有记录 + 当前连续天数（不重复创建）
- 不会重复检查徽章（避免日志噪点）

**徽章自动判定引擎**（`BADGE_RULES`）：

| 徽章 | 图标 | 触发条件 | 判定时机 |
|------|------|----------|----------|
| 🎯 初次打卡 | 完成首次打卡 | totalCheckins ≥ 1 | 打卡后 |
| 🔥 连续三天 | 连续打卡3天 | streak ≥ 3 | 打卡后 |
| ⭐ 坚持一周 | 连续打卡7天 | streak ≥ 7 | 打卡后 |
| 🏅 两周达人 | 连续打卡14天 | streak ≥ 14 | 打卡后 |
| 👑 月度之星 | 连续打卡30天 | streak ≥ 30 | 打卡后 |
| 📝 记录达人 | 累计记录10个任务 | totalTasks ≥ 10 | 创建任务后 |
| 🧠 自我认知 | 完成5次归因分析 | totalAnalyses ≥ 5 | 完成分析后 |

徽章发放为**幂等**：已获得的徽章不会重复发放。

### 2.4 仪表盘数据聚合 — [services/dashboardService.ts](backend/src/services/dashboardService.ts)

一次性返回 Home 仪表盘和 Profile 页面所需的全部聚合数据。

**DashboardData 结构**：

```typescript
interface DashboardData {
  todayTaskCount: number              // 今日任务数
  weekTaskCount: number               // 本周任务总数
  streak: number                      // 连续打卡天数
  weekCheckinDays: number             // 本周已打卡天数
  totalTasks: number                  // 总任务数
  totalCheckins: number               // 总打卡数
  recentProcrastinationType: string   // 最近拖延类型（用于首页展示）
  typeDistribution: Record<string, number> // 拖延类型分布（饼图）
  weekDailyTaskCounts: Array<{date, count}> // 本周每日任务数（折线图）
  weekDailyCheckinCounts: Array<{date, count}> // 本周每日打卡数（趋势图）
}
```

**并行查询优化**：所有独立查询通过 `Promise.all` 并行执行，单次请求响应时间 ~50ms。

## 五、关键技术决策

### 5.1 连续天数计算策略

选择"向前追溯"而非"存储 streak 值"：
- ✅ 无需每日跑定时任务更新所有用户
- ✅ 容错性强：某天漏了重新打卡后 streak 自动恢复
- ✅ 计算简单：只需按天回查 checkin 表

### 5.2 徽章判定时机

采用"事件驱动"而非"定时批处理"：
- ✅ 即时反馈：用户完成操作后立刻获得徽章
- ✅ 实现简单：无需额外的调度框架
- ✅ 异步不阻塞：fire-and-forget 不影响核心 API 响应

### 5.3 仪表盘数据聚合

采用"单次请求批量返回"而非"多个接口分散请求"：
- ✅ 减少前端网络往返
- ✅ 并行查询保证性能
- ✅ 数据结构清晰，前端直接绑定图表
