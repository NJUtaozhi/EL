## 一、前置认知：评委到底在看什么

交互组的评分标准虽然没有公开的详细 rubric，但从比赛定位"**开放式创意赛道，注重作品的创意表达与落地能力**"可以明确：

### 评委心理画像

| 评委类型 | 关注点 | 你的应对 |
|----------|--------|----------|
| **技术评委**（计科/软院老师） | 架构是否合理、技术选型是否有道理、代码质量 | 后端同学主答 |
| **产品评委**（可能来自企业/设计背景） | 是否解决真实痛点、交互是否流畅、UI是否好看 | 前端同学主答 |
| **路人评委**（其他院系老师） | 能不能看懂你的项目在干什么、Demo是否直观 | 演示视频 + PPT 封面一页讲清 |

### 四个核心评估维度（参考专项组评分标准反推）

| 维度 | 权重感 | 核心问题 | 证明方式 |
|------|--------|----------|----------|
| **创新性与创意度** | ~25% | "为什么是这个作品？" | PPT 封面页讲清痛点 + 对比现有方案 |
| **技术实现与复杂度** | ~30% | "它是怎么跑起来的？" | 架构图 + API 调用链路 + AI 能力展示 |
| **实用价值与落地性** | ~25% | "它好用吗？能长期用吗？" | 演示视频展示完整流程 + 用户故事 |
| **用户体验与交互设计** | ~20% | "用起来顺手吗？好看吗？" | UI 截图 + 现场 Demo + 设计理念说明 |

> 🔑 **关键认知**：交互组不是算法组，不需要你发明新模型。评委要看到的是一个**完整、可用、有创意**的产品。工程完成度 > 技术炫酷度。

## 二、前端同学需要掌握到什么程度

### 答辩中前端会被问到的问题（按概率排序）

#### 🔴 必答题（90%会被问到）

**Q1: "你们的前端技术栈是什么？为什么选这个？"**

你必须能回答：
- 用的是 React + TypeScript，因为组件化开发适合复杂交互状态管理
- UI 库选了 TDesign（腾讯企业级设计体系），因为它的组件覆盖面全（Form/DatePicker/Tabs/Progress），且支持主题定制
- 可视化用 ECharts，因为它的图表类型最全（热力图、词云、趋势曲线都能做）
- 状态管理用 Zustand 而不是 Redux，因为项目规模适中，Zustand 更轻量、TS 类型推断更好

**Q2: "你们的页面有哪些？各页面之间的数据流是怎样的？"**

你必须能画出（口述也可以）：
```
用户登录 → Home仪表盘（汇总数据）
         ├─ Log日志页（新建任务）→ Analysis归因页（查看分析结果）
         ├─ Intervention干预页（查看建议卡片）
         ├─ Checkin打卡页（每日打卡）
         └─ Profile画像页（可视化报告）
```

**Q3: "AI 归因结果是怎么在前端展示的？用户等待分析时怎么处理？"**

你必须能回答：
- 分析请求是异步的（调 DeepSeek 需要 3-8 秒），前端做了 loading 骨架屏
- 结果返回后展示：拖延类型（中文标签 + 图标）+ 置信度（进度条）+ 关键词（标签云）+ 建议（卡片列表）
- 如果分析失败，有错误提示 + 重试按钮
- 历史分析结果存储在状态管理中，切换页面不丢失

#### 🟡 高频题（50%概率）

**Q4: "你们的响应式是怎么做的？适配了哪些设备？"**

回答要点：
- CSS 媒体查询分三个断点：手机竖屏（<768px）、平板（768-1024px）、桌面（>1024px）
- 移动端优先设计：底部 TabBar 导航、卡片式布局、触控友好的按钮尺寸
- TDesign 组件本身支持响应式，Grid 系统做了 12 列弹性布局

**Q5: "语音输入是怎么实现的？兼容性怎么样？"**

回答要点：
- 使用浏览器 Web Speech API（SpeechRecognition）
- Chrome/Edge 完美支持，Safari 部分支持，Firefox 不支持（做了能力检测 + 降级方案：显示文本框手动输入）
- 长按录音按钮 → 实时波形动画 → 松开后转文字填入表单

**Q6: "ECharts 图表你们用了哪些？为什么要用热力图？"**

回答要点：
- 饼图：拖延类型分布（直观展示占比）
- 热力图：横轴星期 × 纵轴时段 × 颜色深浅 = 拖延频率（一眼看出"周四下午最容易拖延"）
- 折线图：干预前后拖延次数趋势对比（证明"我们的干预有效"）
- 标签云：拖延关键词可视化
- 所有图表使用治愈系色板（莫兰迪色），统一视觉风格

#### ⚪ 进阶题（被追问时）

**Q7: "如果用户网络不好，前端怎么处理？"**
- API 请求都有超时设置（15秒）
- 失败重试机制（最多3次，指数退避）
- 关键数据做了 localStorage 缓存（用户信息、最近一次分析结果）
- ErrorBoundary 包裹每个页面，避免白屏

**Q8: "你们的代码是怎么组织的？组件复用了哪些？"**
- 目录结构：pages（页面级）/ components（公共组件）/ hooks（自定义Hook）/ store（状态）/ api（请求层）/ utils（工具）/ types（类型）
- 复用组件举例：TaskCard 在 Log 页、Home 页、Analysis 页都用了；EmotionPicker 在 TaskForm 和 Chat 页复用
- 自定义 Hook：useVoiceInput、useEmotion、useCheckin 各自封装独立逻辑

### 前端同学的掌握程度自检清单

| 能力项 | 最低要求（能答辩） | 冲高分要求 |
|--------|-------------------|-----------|
| **React 基础** | 能用函数组件 + Hooks 写页面 | 能解释为什么用 useMemo/useCallback 优化 |
| **TypeScript** | 能用 interface 定义 Props 类型 | 能写出泛型 Hook，类型体操不慌 |
| **状态管理** | 知道 Zustand 怎么用 | 能解释为什么选 Zustand 不选 Redux |
| **ECharts** | 能按文档复制粘贴改配置 | 能讲清楚每个图表的交互设计意图 |
| **CSS/响应式** | 能写出 Flex 布局 | 能讲出移动端优先的设计思路 |
| **API 对接** | 能用 axios 发请求拿数据 | 能讲清楚错误处理、重试、缓存策略 |
| **语音 API** | 能调起浏览器语音识别 | 能讲兼容性矩阵和降级方案 |
| **项目全貌** | 知道自己写了哪些页面 | 能手绘数据流图 + 讲清每个页面的设计意图 |

> 🔑 **一句话**：前端同学要能做到 **"任何一个页面，我都能讲清楚：它做什么、数据从哪来、为什么这样设计"**。

## 三、后端同学需要掌握到什么程度

### 答辩中后端会被问到的问题

#### 🔴 必答题

**Q1: "你们后端的技术架构是什么？为什么选这个技术栈？"**

你必须能回答：
- Node.js + Express，因为：（1）团队前端也用 JS/TS，语言统一降低协作成本；（2）Express 生态成熟，中间件丰富；（3）项目是 I/O 密集型（调 LLM API），Node.js 的异步非阻塞模型天然适合
- 数据库选 MySQL + Prisma ORM，因为：（1）用户-任务-分析的关系型数据天然适合关系型数据库；（2）Prisma 提供类型安全的数据库操作，避免 SQL 注入；（3）迁移管理方便，适合迭代开发
- AI 能力接 DeepSeek API，因为：（1）性价比高（百万 token 1元）；（2）中文能力强；（3）支持 JSON 模式输出，方便解析归因结果

**Q2: "数据库表结构是怎样的？表之间的关系是什么？"**

你必须能画出（口述）：
```
User (1) ──── (N) Task       用户有多个任务
Task (1) ──── (1) Analysis   每个任务对应一次分析
User (1) ──── (N) Checkin    用户有多条打卡记录
User (1) ──── (N) Badge      用户有多个徽章
```

每张表的核心字段要能说出来。

**Q3: "DeepSeek 归因分析是怎么实现的？Prompt 是怎么设计的？"**

这是最核心的问题，必须详细回答：

```
调用流程：
1. 用户提交任务（任务名 + 拖延理由 + 情绪标签 + 学期阶段）
2. 后端组装 System Prompt（拖延心理学专家角色 + 5种拖延类型定义）
3. 将用户信息拼接为 User Message
4. 调用 DeepSeek API（model: deepseek-chat, temperature: 0.3）
5. 解析返回的 JSON：{ type, confidence, keywords, suggestion }
6. 存入 Analysis 表，返回前端展示

Prompt 设计要点：
- System Prompt 明确角色（"你是拖延心理学专家"）
- 给出明确的分类标准（5种类型 + 每种的典型表现）
- 要求 JSON 格式输出（便于解析）
- Temperature 设低（0.3），保证输出稳定
- 设置 max_tokens 限制，控制成本
```

**Q4: "API 接口是怎么设计的？有哪些接口？"**

你必须能列出核心接口：
```
POST   /api/tasks          创建任务
GET    /api/tasks           获取任务列表
DELETE /api/tasks/:id       删除任务
POST   /api/analysis/:taskId  触发归因分析
GET    /api/analysis/:taskId  获取分析结果
POST   /api/checkin         打卡
GET    /api/checkin/history  打卡历史
GET    /api/user/profile     用户画像聚合数据
```

返回格式统一为 `{ code: 0, data: {...}, message: "ok" }`

#### 🟡 高频题

**Q5: "用户认证是怎么做的？安全吗？"**

回答要点：
- JWT 认证：登录后签发 token（7天过期），前端存 localStorage，每次请求带 Authorization header
- 密码用 bcrypt 哈希存储，不存明文
- 中间件验证 token，注入 req.user 供后续使用
- API 做了 CORS 白名单限制

**Q6: "如果 DeepSeek API 挂了或者超时了，后端怎么处理？"**

回答要点：
- 设置了 30 秒超时
- 失败后有重试机制（最多2次）
- 重试仍失败则返回友好的错误信息给前端
- 同时记录错误日志（Winston），方便排查
- 不会让整个服务崩溃（try-catch + 全局错误处理中间件）

**Q7: "后端做了哪些中间件？"**

回答要点：
- `auth.ts`：JWT 认证中间件
- `errorHandler.ts`：全局错误处理，统一返回格式
- `validator.ts`：Zod 请求参数校验（防止无效数据进入数据库）
- CORS：限制允许的前端域名
- 请求日志：记录每个请求的方法、路径、耗时

#### ⚪ 进阶题

**Q8: "如果用户量变大，这个架构有什么瓶颈？你会怎么优化？"**
- 当前瓶颈：每次归因分析都调 DeepSeek API（费钱 + 慢）
- 优化方向：对相似任务做结果缓存（Redis）；引入任务队列避免阻塞；对高频查询加数据库索引
- 数据库：单表数据量大时可以分表（按 userId 哈希）

**Q9: "为什么不用 Python/Flask 而用 Node.js？"**
- Node.js 的异步非阻塞模型更适合 I/O 密集型场景（大量 LLM API 调用）
- 团队技术栈统一（前端也是 TS），降低协作成本
- Express 生态成熟，中间件丰富

### 后端同学的掌握程度自检清单

| 能力项 | 最低要求（能答辩） | 冲高分要求 |
|--------|-------------------|-----------|
| **Express** | 能写路由 + 控制器 + 中间件 | 能解释中间件执行顺序和洋葱模型 |
| **Prisma/MySQL** | 能建表 + CRUD | 能解释为什么用 Prisma 而不是原生 SQL |
| **LLM API 调用** | 能调通 DeepSeek | 能讲清楚 Prompt 工程的设计思路 |
| **认证授权** | 能用 JWT 做登录 | 能讲清楚 JWT vs Session 的取舍 |
| **错误处理** | 能 try-catch | 能讲全局错误处理中间件的设计 |
| **API 设计** | 能写 RESTful 接口 | 能讲清楚 URL 命名规范和状态码选择 |
| **Docker/部署** | 能写 docker-compose.yml | 能讲清楚多容器编排和网络配置 |
| **项目全貌** | 知道所有接口的功能 | 能手绘架构图 + 数据流图 + 讲清每个技术选型的理由 |

> 🔑 **一句话**：后端同学要能做到 **"任何一个技术决策，我都能讲清楚：为什么选它、它解决了什么问题、还有什么替代方案"**。

## 四、两人协作黄金法则

### 4.1 互相了解对方的领域

**答辩中最致命的场景**：评委问前端同学一个偏后端的问题，前端完全答不上来；反之亦然。

**最低要求**（两人都要能做到）：

| 前端必须了解后端 | 后端必须了解前端 |
|-----------------|-----------------|
| 知道有哪些 API 接口，URL 是什么 | 知道有哪些页面，每个页面干什么 |
| 知道 DeepSeek 归因的大致流程（不是调前端直接调的） | 知道前端状态管理用什么（Zustand）|
| 知道数据库有哪些表（User/Task/Analysis/Checkin/Badge） | 知道 ECharts 用了哪几种图表 |
| 能说出架构图：前端 → Express → DeepSeek / MySQL | 能说出页面间的导航关系 |
| 知道 JWT 认证的流程（登录拿 token → 每次请求带 token） | 知道语音输入是怎么实现的 |

### 4.2 答辩分工策略

**场景一：评委问"介绍一下你们的项目"**
- 前端同学先讲（1分钟）：产品定位 + 解决了什么问题 + 页面展示
- 后端同学接着讲（1分钟）：技术架构 + AI 能力 + 数据流

**场景二：技术深挖**
- 技术实现细节 → 后端同学主答
- UI/UX、交互设计 → 前端同学主答
- 关于 AI/LLM 的问题 → 后端同学主答（Prompt 设计、API 调用）
- 关于选题创意、用户场景 → 前端同学主答（更贴近用户视角）

**场景三：一个人被问住了**
- 队友可以补充，但要自然（"我补充一下"而不是"他说错了"）
- 提前约定信号：比如碰一下队友的手肘 = "你来接"

### 4.3 答辩前互相考核

答辩前一周，两人互相模拟评委提问：

```
前端考后端：
"你们的Prompt是怎么设计的？"
"为什么不用MongoDB？"
"DeepSeek返回格式不对怎么办？"

后端考前端：
"Zustand 和 Redux 有什么区别？"
"移动端适配你是怎么做的？"
"ECharts 热力图的配置项有哪些关键参数？"
```

## 五、答辩PPT模板

建议 PPT 控制在 **12-15 页**，答辩时间通常 5-8 分钟（陈述）+ 3-5 分钟（提问）。

### 推荐结构

| 页码 | 内容 | 时间 | 讲者 |
|------|------|------|------|
| 1 | **封面**：作品名 + 团队名 + 一句话slogan | 15秒 | A |
| 2 | **痛点**：拖延有多普遍（数据）+ 现有方案的问题 | 30秒 | A |
| 3 | **解决方案**：我们的产品是什么 + 核心差异化 | 30秒 | A |
| 4 | **核心功能一览**：功能模块图（日志→归因→画像→干预→打卡） | 45秒 | A |
| 5 | **技术架构**：前端→后端→AI→数据库架构图 | 45秒 | B |
| 6 | **AI 归因引擎**：DeepSeek 调用链路 + Prompt 设计 | 45秒 | B |
| 7 | **拖延画像与可视化**：热力图 + 趋势图 + 标签云截图 | 30秒 | A |
| 8 | **干预建议系统**：建议卡片 + 打卡机制 | 30秒 | B |
| 9 | **用户流程演示**：3张截图串联完整使用流程 | 30秒 | A |
| 10 | **技术亮点**：语音输入 / 响应式 / 错误处理 / 部署方案 | 45秒 | B |
| 11 | **项目进度**：已完成什么 + 后续计划 | 30秒 | B |
| 12 | **演示视频**：播放2-3分钟（或放在答辩前播放） | 2-3min | - |
| 13 | **总结与展望**：核心价值 + 后续迭代方向 | 30秒 | A |

### PPT 制作要点

- **少字多图**：每页不超过 5 行文字，多用截图、架构图、图表
- **颜色统一**：使用莫兰迪/治愈色系，和产品 UI 风格一致
- **动画克制**：不要花里胡哨的切换动画
- **关键数字要大**：比如"覆盖5种拖延类型""连续打卡30天"
- **备一页"备用幻灯片"**：放详细技术细节，被追问时翻出来

## 六、演示视频拍摄指南

### 6.1 视频规格
- 时长：**2-3 分钟**（不要超过，评委没耐心）
- 分辨率：1080P 以上
- 格式：MP4
- 配音：必须有解说（清晰、自然、不要念稿感）

### 6.2 视频结构（3分钟版）

| 时间段 | 内容 | 画面 |
|--------|------|------|
| 0:00-0:20 | **开场**：痛点引入 + 作品名 | 文字动画 + 生活场景（比如一个人对着电脑发呆） |
| 0:20-0:40 | **注册登录** | 屏幕录制：打开网页 → 登录 → 进入首页 |
| 0:40-1:20 | **核心流程**：记录拖延 → AI分析 | 屏幕录制：新建任务（语音输入）→ 触发分析 → 看到归因结果 |
| 1:20-1:50 | **可视化**：拖延画像 | 屏幕录制：Profile 页面，展示热力图、趋势图、标签云 |
| 1:50-2:20 | **干预 + 打卡** | 屏幕录制：查看建议卡片 → 收藏 → 打卡 |
| 2:20-2:50 | **效果展示** | 对比：干预前 vs 干预后的拖延频率下降 |
| 2:50-3:00 | **结尾**：作品名 + 团队名 + 感谢 | 定格画面 |

### 6.3 拍摄技巧

- 🎙️ **配音**：用剪映/PR 录音，语速适中（约 180-200 字/分钟），加背景音乐（轻量、不抢人声）
- 📱 **屏幕录制**：用 OBS 录制，分辨率 1080P，鼠标移动平滑，不要有桌面图标/通知弹窗
- ✨ **后期**：关键操作加放大镜效果、文字标注；页面切换加简单转场
- ❌ **避免**：录制时不要加载转圈太久（提前打开页面缓存好）；不要有报错画面

## 七、答辩现场话术策略

### 7.1 开场白（前30秒定调）

> "大家好，我们是XX团队。我们的作品是「不拖延实验室」—— 一个不只是告诉你'为什么拖延'，而是真正帮你**减少拖延**的智能拖延管理系统。"

✅ **好的开场**：一句话讲清做什么 + 差异化 + 自信但不傲慢
❌ **差的开场**："我们做了一个小程序..."（太平淡）；"众所周知拖延是一个普遍存在的问题..."（太啰嗦）

### 7.2 技术表述的黄金句式

| 场景 | 话术模板 |
|------|----------|
| 讲架构 | "我们的架构分为三层：前端展示层用 React，后端服务层用 Node.js，AI 能力层接 DeepSeek API。" |
| 讲AI | "我们让大模型扮演拖延心理学专家的角色，通过精心设计的 Prompt 引导它分析用户的拖延行为。" |
| 讲数据 | "从数据流的视角看，用户输入经过后端组装 Prompt → 调用 DeepSeek → 解析 JSON 结果 → 存入数据库 → 返回前端可视化。" |
| 讲难点 | "这个功能的挑战在于XXX，我们的解法是YYY。" |
| 讲选型 | "我们对比了 A/B/C 三个方案，最终选 B 是因为..." |

### 7.3 应对刁钻问题的策略

**策略一：遇到不会的问题**
> "这是一个很好的问题。我们目前的做法是XXX，关于您提到的这个方向，我们也正在调研中，计划在后续版本中实现。"

**策略二：评委质疑你的设计选择**
> "我们确实考虑过您提到的方案。它的优势是XXX，但我们认为在当前场景下，选择YYY更合适，因为..."

**策略三：评委指出Bug/不足**
> "感谢老师的建议！这确实是我们接下来要优化的点。我们会在XXX方面继续改进。"

**策略四：评委问了一个你队友更懂的问题**
> "关于技术实现细节，让我队友来补充一下。"（自然地递话筒）

### 7.4 绝对不要做的事

- 🙅 和评委争论（"老师你理解错了"）
- 🙅 说"这个很简单"（贬低自己的工作）
- 🙅 说"我们用了最新的XXX技术"但实际没深入用
- 🙅 两个人抢话说
- 🙅 念 PPT（PPT 是给评委看的，你的嘴应该讲 PPT 上没有的内容）
- 🙅 说"时间不够所以没做完"（展示已完成的，不要强调未完成的）

## 八、常见扣分项与避坑清单

### 8.1 项目文档扣分项

| 扣分项 | 避坑方法 |
|--------|----------|
| 没有仓库链接 | README 第一行就放 GitHub/njubox 链接 |
| 没有部署指南 | 写清楚 `npm install && npm run dev` 能跑起来 |
| 没有使用说明 | 截图 + 步骤说明用户怎么用 |
| 没有分工说明 | 写清楚谁做了什么 |
| 没有技术选型理由 | 每个技术选择后面加一句话理由 |
| 文档里写了但实际没实现 | 诚实标注"计划中"，不要虚假宣传 |

### 8.2 演示视频扣分项

| 扣分项 | 避坑方法 |
|--------|----------|
| 超过3分钟 | 严格控制，剪掉所有废话镜头 |
| 没有声音/声音太小 | 配音后检查音量 |
| 画面模糊 | OBS 录制设置 1080P |
| 加载转圈太久 | 提前打开所有页面，Demo 时直接展示 |
| 只展示了界面没有展示功能 | 必须走完一个完整用户故事 |
| 视频结尾没有作品名和团队名 | 最后 5 秒放定格画面 |

### 8.3 答辩现场扣分项

| 扣分项 | 避坑方法 |
|--------|----------|
| 超时被喊停 | 提前排练，卡时间（陈述不超过6分钟） |
| Demo 翻车（报错/加载不出来） | 提前准备好本地环境，断网也能跑 |
| 两个人配合生硬 | 提前演练 3-5 次，磨合话术切换 |
| PPT 和 Demo 切换卡顿 | 一个人控制电脑，提前打开所有窗口 |
| 回答问题时眼神游离 | 看着提问的评委回答，队友点头附和 |
